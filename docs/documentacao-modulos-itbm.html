Visão geral da plataforma

O ITBM (produto comercial G360 Enterprise) é uma plataforma de gestão empresarial multi-tenant: cada cliente possui schema PostgreSQL dedicado; o catálogo de tenants e configurações globais ficam no schema public.

A API REST é servida em /api/v1 (Express). O frontend (React 19, Vite 6, MUI 7) consome essa API com Authorization: Bearer e, quando aplicável, X-Tenant-Slug. O Prisma resolve o cliente correto por tenant em cada requisição.

RBAC: perfis possuem permissões módulo + ação (ex.: HELPDESK/READ). Perfis administrativos podem ignorar checagens finas. Escopo: gestores veem dados dos centros de custo sob sua responsabilidade; demais usuários, em geral, apenas o próprio escopo (ver access-scope no backend).

Governança: trilho de auditoria (AuditLog), sanitização de dados sensíveis, export CSV para compliance, retenção configurável, rate limit por utilizador quando autenticado, e notificações em tempo real via SSE com JWT dedicado (não usar o access token na query string).

Em cada módulo abaixo, Regras de negócio descrevem o comportamento implementado no backend (validações, SLAs, transições). Manual de funcionamento orienta o utilizador ou administrador (passos na UI). Os identificadores RN-XXX-NN facilitam rastreio; a referência canónica técnica continua em DOCUMENTACAO.md e em BACKEND/src/services.

Autenticação, sessão e multi-tenant

Função

Identificar o usuário, emitir JWT (com schemaName do tenant), refresh token e aplicar middleware de tenant em todas as rotas protegidas.

Frontend

Páginas de login e callback Azure (quando habilitado); armazenamento de tokens e slug; interceptor Axios para refresh em 401.

Backend

Rotas em /api/v1/auth, /users, /roles, /tenants (administração de empresas no contexto autorizado).

Segurança

  • Helmet, CORS, rate limit em /api, sanitização de body (XSS).
  • Resolução de tenant: JWT → header x-tenant-slug → fallback.
  • POST /auth/stream-token (Bearer) — emite JWT curto purpose=sse para EventSource em notificações.
  • PATCH /auth/me/notification-preferences — preferências de canais/categorias (JSON em User).

RBAC (módulos relacionados)

CONFIG (usuários/organização), SUPER_ADMIN (plataforma global quando aplicável).

Regras de negócio

RN-AUTH-01 Cada requisição autenticada deve carregar JWT válido; o token inclui o schemaName do tenant para o Prisma apontar ao schema PostgreSQL correcto.

RN-AUTH-02 Resolução de tenant: JWT → cabeçalho X-Tenant-Slug → cliente padrão (public) quando aplicável.

RN-AUTH-03 Em 401, o frontend renova o access token via refresh token; sessão inválida redirecciona ao login.

RN-AUTH-04 Super Admin no contexto public acede ao menu de plataforma (tenants); dentro de um tenant, mantém bypass de permissões conforme perfil.

RN-AUTH-05 Eventos de sessão gravados em AuditLog (módulo AUTH): AUTH_LOGIN (local/Azure/LDAP), AUTH_REFRESH, AUTH_LOGOUT — sem expor segredos no newData.

RN-AUTH-06 O stream SSE de notificações só aceita na query ?token= um JWT com purpose: 'sse' emitido por POST /auth/stream-token; o access token completo não deve ser usado na URL.

Manual de funcionamento

  1. Abrir o URL da aplicação e, se necessário, escolher o tenant (slug) acordado com a organização.
  2. Autenticar com credenciais locais ou fluxo Azure AD, conforme o ambiente.
  3. Após o login, o menu lateral mostra apenas módulos para os quais o seu perfil tem permissão de leitura (ou equivalente).
  4. Para terminar sessão, use Sair no menu do utilizador (canto superior).

Segurança, auditoria e governança

Função

Proteção em camadas (HTTP, limite de pedidos, sanitização), trilho de auditoria para alterações e eventos de sessão, exportação para compliance e política de retenção de registos históricos.

Backend (ficheiros de referência)

  • middlewares/audit.middleware.js — registo após resposta 2xx; derivação de ação HTTP; integração com audit-sanitize.util.js.
  • utils/audit-log-scope.util.js — mesmo escopo RBAC para listagem e export de AuditLog.
  • middlewares/rate-limit.middleware.js — loginLimiter, globalLimiter (chave por utilizador JWT ou IP), strictLimiter, referenceLimiter.
  • middlewares/security-logger.middleware.js — eventos 401/403/429 via Winston.
  • services/cron.service.js — pruning de AuditLog (agendamento diário).

API

GET /audit-logs — listagem paginada; filtros module, start, end.
GET /audit-logs/export — CSV (mesmos filtros; limite máximo configurável por ambiente).
Permissão: ACTIVITY_LOG / READ.

Variáveis de ambiente (auditoria)

  • AUDIT_LOG_RETENTION_DAYS — dias a manter (default 365; 0 não apaga via job).
  • AUDIT_LOG_RETENTION_DISABLED=true — desliga o job de pruning.
  • AUDIT_LOG_EXPORT_MAX — máximo de linhas no CSV export.
  • JWT_SSE_EXPIRES_IN — TTL do token só para SSE (ex.: 10m).
  • RATE_LIMIT_BYPASS=true — testes de carga (sem limites).

Regras de negócio

RN-SEC-01 O middleware audit() só grava quando existem req.user.userId e req.prisma; payloads sensíveis são mascarados antes de persistir.

RN-SEC-02 res.json e res.send são intercetados de forma a evitar duplicar linhas de auditoria quando o Express encadeia ambos.

RN-SEC-03 Chamados e problemas (Help Desk) registam mutações com módulos HELPDESK e PROBLEM respectivamente.

RN-SEC-04 O limite global de API usa chave uid:<userId> quando o Bearer é válido, reduzindo colisão entre utilizadores no mesmo IP.

RN-SEC-05 Export CSV inclui BOM UTF-8 e colunas para análise forense; o escopo de linhas segue o mesmo critério de gestor/equipa que a listagem.

Manual de funcionamento

  1. Administradores de compliance: use Atividades / auditoria no front (/activities) e o export CSV quando precisar de ficheiro externo.
  2. Defina AUDIT_LOG_RETENTION_DAYS em função da política da organização e requisitos legais.
  3. Em incidentes de segurança, cruze logs/security.log e logs/app.log (Winston) com entradas de AuditLog no PostgreSQL.

Função

Frontend

/projects, /projects/:id, /projects/portfolio, /projects/team-status-report.

Backend

/projects, /project-tasks.

Regras relevantes

  • Fluxos de aprovação de baseline e custos (ligação à esteira de aprovações).
  • Propostas e custos com estados (rascunho → aprovação).
  • Escopo de dados pode respeitar centro de custo / perfil.

RBAC

Módulo PROJECTS: READ, CREATE, WRITE, DELETE, DELETE_PROJECT, APPROVE_BASELINE, APPROVE_FINANCIAL.

Ligações

GMUD opcional por projeto; tickets podem referenciar projeto; financeiro via centro de custo e custos.

Regras de negócio

RN-PRJ-01 Projetos estão ligados a departamento, centro de custo, gerente e tech lead; custos e propostas podem exigir aprovação na esteira (/approvals).

RN-PRJ-02 Listagens e detalhes podem ser filtrados por access-scope: administradores sem filtro de CC; gestores veem utilizadores dos CC que gerem; restantes, escopo próprio.

RN-PRJ-03 GMUD pode ser associada ao projeto (projectId); tickets podem referenciar linkedProjectId para rastreio de demanda.

Manual de funcionamento

  1. Lista: em /projects consulte o portfólio; use filtros conforme a UI.
  2. Novo projeto: crie o registo, preencha dados de negócio, CC e responsáveis.
  3. Detalhe: em /projects/:id registe membros, atas, follow-ups, propostas, custos e riscos do projeto.
  4. Aprovações: quando o estado pedir aprovação de baseline ou financeiro, o aprovador trata em Minhas Aprovações.
  5. Relatórios: use /projects/portfolio ou /projects/team-status-report para visões consolidadas.

Gestão de mudança (GMUD / RDM)

Função

RFC, questionário de risco, aprovadores (CAB), serviços/ativos afetados, calendário, janelas de congelamento, PIR ao falhar/cancelar.

Frontend

/changes.

Backend

/changes, /change-templates, /freeze-windows.

Regras

  • riskAssessment alimenta nível de risco calculado.
  • Mudanças não emergenciais não podem violar FreezeWindow ativa.
  • Transição para FAILED/CANCELLED após estados operacionais exige causa raiz e lições aprendidas (PIR).
  • Vínculo com incidentes e tickets (escalação).

RBAC

Módulo GMUD: READ, CREATE, WRITE, DELETE, EDIT_CHANGE, SCHEDULE, APPROVE, ATTACH.

Jobs

Cron de SLA de tickets interage com operações de TI; GMUD usa dados de schedule conflict conforme serviço.

Regras de negócio

RN-GMUD-01 O nível de risco é calculado a partir do questionário riskAssessment (produção, downtime, testes, rollback): pontuação agregada → BAIXO / MEDIO / ALTO / CRITICO.

RN-GMUD-02 Mudanças com tipo diferente de EMERGENCIAL não podem ter janela agendada sobreposta a uma FreezeWindow activa (validação em criação e actualização).

RN-GMUD-03 PIR: ao passar para FAILED ou CANCELLED depois de estados operacionais (APPROVED, APPROVED_WAITING_EXECUTION, EXECUTED), são obrigatórios causa raiz e lições aprendidas.

RN-GMUD-04 Estados válidos incluem rascunho, aprovação, execução, rejeição, revisão solicitada, etc., conforme validador Yup no backend.

RN-GMUD-05 Ligação a incidentes (relatedIncidentId) e tickets para rastreio de causa e escalação.

Manual de funcionamento

  1. Em /changes, crie uma nova RFC ou use um template (se disponível para o seu perfil).
  2. Preencha justificativa, impacto, tipo, datas planeadas e questionário de risco.
  3. Submeta para aprovação; aprovadores (CAB) registam voto e comentários.
  4. Respeite o calendário: o sistema bloqueia agendamento em períodos de congelamento (exceto emergencial).
  5. Após execução, preencha planos de execução/teste e encerramento; se a mudança falhar ou for cancelada após aprovação/execução, preencha o PIR (causa raiz e lições aprendidas).

Incidentes (ITIL)

Função

Registro de incidentes com prioridade (impacto × urgência), SLA de resposta/resolução, histórico, comentários, anexos.

Frontend

/incidents.

Backend

/incidents.

Integrações

Ativo relacionado; mudança relacionada; base de conhecimento no processo operacional.

RBAC

Módulo INCIDENT: READ, WRITE, DELETE, VIEW_ALL_CC, ESCALATE.

Regras de negócio

RN-INC-01 A prioridade (P1–P4) é calculada pela matriz Impacto × Urgência (valores normalizados para maiúsculas); combinação desconhecida assume P3.

RN-INC-02 SLA: P1 — resposta 60 min / resolução 240 min; P2 — 240/480; P3 — 480/1440; P4 — 1440/4320 (minutos a partir da criação).

RN-INC-03 Violação de SLA: se não houver respondedAt e o tempo actual ultrapassar slaResponseDue, ou se não resolvido e ultrapassar slaResolveDue, considera-se violação na lógica de verificação.

RN-INC-04 Código único gerado pelo repositório; colisão implica nova tentativa.

RN-INC-05 Histórico de alterações e auditoria registados na criação e eventos relevantes.

Manual de funcionamento

  1. Aceda a /incidents e registe um novo incidente com título, descrição, categoria e classificação de impacto e urgência.
  2. O sistema calcula a prioridade e os prazos de SLA; acompanhe o relógio na lista ou detalhe.
  3. Associe opcionalmente ativo ou mudança relacionada para contexto de CMDB/RDM.
  4. Atualize estado, responsável e comentários até resolução; use escalação conforme processo interno (permissão ESCALATE).

Help Desk, Portal, Service Desk e problemas

Função

Portal (/portal): solicitante abre chamados via catálogo (wizard), acompanha lista. Service Desk (/servicedesk): fila do agente, detalhes, mensagens, SLA. Catálogo (/servicedesk/catalog): serviços, categorias, formulários JSON (formSchema). Problemas ITIL (/servicedesk/problems): Problem Management.

Backend

/tickets, /service-catalog, /sla-policies, /problems, /helpdesk-config, /support-groups.

Entidades-chave

  • Ticket: categoria, serviço, respostas customizadas, SLA, pausa (slaPausedAt, minutos pausados), mensagens, vínculos (ativo, projeto, mudança, problema).
  • ServiceCatalog / SlaPolicy: definem tempos e formulários dinâmicos.

Automação

Cron (~5 min) verifica SLA; IMAP para e-mails de resposta em tickets (quando configurado).

RBAC

Módulos HELPDESK (ações CREATE, VIEW_QUEUE, UPDATE_STATUS, escalações, etc.) e PROBLEM (READ, DECLARE, MASS_RESOLUTION, LINK_ORPHANS).

Regras de negócio

RN-HD-01 Código do chamado: formato compacto HD + dois dígitos do ano + sequência de 4 dígitos (ex.: HD260359), sem hífens; sequência atómica por ano civil (PostgreSQL). Respostas por e-mail com o código antigo HD-AAAA-NNNN no assunto continuam a ser associadas ao chamado.

RN-HD-02 SLA: minutos de resposta/resolução vêm, por ordem, da política ligada ao serviço do catálogo, depois ao grupo de suporte, senão tabela fallback por prioridade (LOW/MEDIUM/HIGH/URGENT).

RN-HD-03 Calendário de SLA: se a configuração do helpdesk activar calendário de negócio, os prazos usam horário útil; caso contrário, minutos de relógio.

RN-HD-04 Ao criar com supportGroupId, se autoAssignOnCreate estiver activo (config), atribuição round-robin entre membros do grupo.

RN-HD-05 Estado WAITING_USER: ao entrar, regista slaPausedAt; ao sair, acumula minutos pausados e desloca slaResponseDue e slaResolveDue pelo tempo pausado.

RN-HD-06 RESOLVED/CLOSED: define resolvedAt na primeira resolução; CLOSED também define closedAt.

RN-HD-07 Cron (~5 min): chamados OPEN/IN_PROGRESS com SLA vencido e ainda sem resposta/resolução marcam slaBreached e notificam o assignee.

RN-HD-08 Portal: listagem como solicitante filtra por requesterId; fila de agente exige permissões HELPDESK adequadas.

RN-HD-09 Problemas (ITIL): módulo PROBLEM — permissões DECLARE, MASS_RESOLUTION, LINK_ORPHANS para gestão de causa raiz e vínculo de tickets órfãos (ver API /problems).

RN-HD-10 Auditoria: operações mutáveis em /tickets e /problems geram entradas em AuditLog (middleware audit) com dados sanitizados.

Manual de funcionamento

Solicitante — Portal (/portal)

  1. Clique em Novo ticket.
  2. Escolha a categoria (passo 1) e o serviço (passo 2); use a busca para filtrar.
  3. Preencha título, descrição, campos obrigatórios do formulário do serviço, ativo opcional e grupo de atendimento se aplicável.
  4. Submeta; receberá e-mail (se configurado) com link para /portal/tickets/:id.
  5. Acompanhe a tabela Meus Chamados; abra o detalhe para ler mensagens da equipa.

Agente — Service Desk (/servicedesk)

  1. Visualize a fila global; assuma ou filtre por grupo/atribuído.
  2. Actualize estado (OPEN → IN_PROGRESS → WAITING_USER / RESOLVED / CLOSED) conforme o processo.
  3. Use aguardando utilizador quando precisar de informação do solicitante — o SLA pausa até retomar.
  4. Mensagens internas vs públicas: respeite permissões VIEW_INTERNAL_NOTES / WRITE_INTERNAL_NOTES.
  5. Escale para problema, mudança ou projeto através das acções de escalação (permissões específicas).

Administrador do catálogo (/servicedesk/catalog)

  1. Mantenha categorias, serviços, políticas de SLA e formulários JSON (formSchema) alinhados ao catálogo ITIL da organização.
  2. Configure grupos de suporte e membros para a atribuição round-robin.

Gestão de problemas (/servicedesk/problems)

  1. Declare problemas conhecidos e associe tickets relacionados (causa comum).
  2. Use resolução em massa e vinculação de órfãos conforme permissões PROBLEM.

Integração entre módulos

Abaixo, visão de como os domínios se apoiam uns nos outros. Os dados compartilhados passam por entidades com FKs ou tabelas de junção no Prisma; regras de negócio ficam nos services do backend e permissões no RBAC.

Regras de negócio transversais

RN-X-01 RBAC: cada rota da API invoca authorize('MÓDULO','AÇÃO') (ou equivalente); a matriz canónica está em rbac-matrix.json.

RN-X-02 Escopo: getUserAccessScope determina CCs acessíveis; administradores do tenant não são filtrados por CC; gestores veem utilizadores dos CC que gerem.

RN-X-03 Multi-tenant: todas as operações de negócio usam o cliente Prisma do schema do tenant corrente (excepto rotas explicitamente globais).

RN-X-04 Anexos: módulos utilizam /uploads com permissão UPLOAD/WRITE onde aplicável.

RN-X-05 E-mail: templates (ex.: abertura de ticket) dependem de MailService e configuração SMTP/integração no tenant.

RN-X-06 Auditoria: operações mutáveis em rotas com audit() geram AuditLog com dados sanitizados; eventos de login/refresh/logout usam caminho dedicado no controller de auth.

RN-X-07 Rate limit: limite global por utilizador autenticado (JWT) ou IP; login limitado por email para mitigar brute-force.

RN-X-08 SSE: notificações em tempo real usam JWT de curta duração com purpose: 'sse', não o access token na URL.

RN-X-09 Jobs: cron em instância única (PM2); inclui SLA de tickets, e-mail IMAP, vencimentos e pruning de auditoria.

RN-X-10 Integrações: /integrations (LDAP, e-mail, Azure) e /reference para dados de apoio à UI; testes de conectividade conforme controlador.

Manual transversal (boa prática)

  1. Garanta que utilizadores têm perfil e centro de custo correctos antes de operar módulos sensíveis (financeiro, incidentes).
  2. Use vínculos (ativo, projeto, mudança) nos chamados e incidentes para relatórios e root cause consistentes.
  3. Para mudanças que afectam produção, alinhe GMUD, janelas de congelamento e comunicação aos solicitantes no Service Desk.

4. Multi-tenant e resolução de tenant

  • Catálogo: modelo Tenant no schema public (slug, schemaName, plano, enabledModules, etc.).
  • tenantResolver() (rotas autenticadas): prioridade — payload JWT (schemaName / tenantSlug) → decodificar Authorization: Bearer se ainda não houver req.user → GET com ?token= apenas para JWT com purpose: 'sse' (SSE de notificações) → header x-tenant-slug → cliente Prisma padrão.
  • tenantResolverPublic(): login/Azure/LDAP — tenant por tenantSlug no body/params/header.
  • tenantResolverAdmin(): operações de super-admin no catálogo (req.prisma = client público).

5. Segurança, governança e conformidade

5.1 Autenticação

  • Login: local (email/senha), Azure AD (MSAL), LDAP — conforme rotas em /auth/login, /auth/azure, /auth/ldap.
  • Tokens: access JWT (payload: userId, email, roles, tenantSlug, schemaName); refresh armazenado na tabela RefreshToken com revogação no logout.
  • Endpoints públicos: POST /auth/login, /auth/azure, /auth/ldap, /auth/refresh, /auth/logout, GET /auth/azure-config/:tenantSlug.
  • Endpoints autenticados: POST /auth/stream-token, GET /auth/me, PATCH /auth/me/notification-preferences.
  • Auditoria de sessão (AuditLog, módulo AUTH): eventos AUTH_LOGIN, AUTH_REFRESH, AUTH_LOGOUT gravados no controller (login bem-sucedido por canal LOCAL/AZURE/LDAP; refresh e logout após validação do refresh token). Não dependem do middleware audit() em rotas sem JWT completo.

5.2 Rate limiting (rate-limit.middleware.js)

Limiter Uso típico Notas
loginLimiter /auth/login, Azure, LDAP Chave por email no body; limite reforçado em produção
globalLimiter Todas as rotas /api Chave uid:<userId> se Bearer JWT válido; senão IP. Health GET /api/health é ignorado
strictLimiter Operações sensíveis (ex.: refresh) Limite mais baixo
referenceLimiter Endpoints de referência leves —

RATE_LIMIT_BYPASS=true desativa limites para testes de carga.

5.3 Proteção HTTP e payload

  • Helmet: CSP, HSTS, noSniff, hidePoweredBy, Referrer-Policy, Permissions-Policy.
  • CORS: origens em desenvolvimento (localhost e IPs privados); em produção ALLOWED_ORIGINS (lista separada por vírgulas).
  • Sanitização global: middleware sanitize em req.body (mitigação XSS).
  • Limites de body: JSON/urlencoded 100kb em app.js.

5.4 Logging de segurança

  • securityLogger: em respostas 401, 403, 429 regista evento via Winston (logger.warn com category: 'SECURITY'), incluindo IP, método, path, utilizador quando existir.
  • Funções auxiliares: logFailedLogin, logBlockedUpload, writeSecurityLog — mesmo canal (sem duplicar appendFile manual).

5.5 Auditoria de dados (AuditLog)

  • Middleware audit(module, action?): interceta res.json / res.send em sucesso 2xx; deriva ação (POST→CREATE, PUT/PATCH→UPDATE, DELETE→DELETE, GET→READ) se omitida; grava oldData/newData sanitizados (audit-sanitize.util.js — passwords, tokens, emails mascarados, profundidade limitada).
  • Requisito: req.user.userId e req.prisma; sem utilizador autenticado não grava (evita linhas inválidas).
  • Entidade: entityId obtido de req.params (id, ticketId, minuteId, costId, etc.) ou corpo.
  • Rotas com auditoria: dezenas de módulos; Help Desk (HELPDESK) e Problemas (PROBLEM) incluem mutações; ver ticket.routes.js e problem.routes.js.
  • Listagem / export: GET /audit-logs e GET /audit-logs/export (CSV) com mesmo filtro RBAC que a listagem (audit-log-scope.util.js). Export: limite AUDIT_LOG_EXPORT_MAX (default 10 000, máx. 100 000), BOM UTF-8.
  • Retenção: cron diário 03:30 — remove registos com createdAt anterior a N dias (AUDIT_LOG_RETENTION_DAYS, default 365). Desativar pruning: AUDIT_LOG_RETENTION_DAYS=0 ou AUDIT_LOG_RETENTION_DISABLED=true.

5.6 Notificações em tempo real (SSE)

  • GET /notifications/stream: Server-Sent Events; não envia o access token completo na query.
  • Fluxo: cliente autenticado chama POST /auth/stream-token → recebe JWT de curta duração com purpose: 'sse' → EventSource usa ?token=<sse_jwt>.
  • Expiração SSE: JWT_SSE_EXPIRES_IN (ex.: 10m).
  • Tenant: tenant-resolver valida o token SSE e preenche req.user para authorize('NOTIFICATIONS', 'READ').

5.7 RBAC e escopo

  • Matriz: rbac-matrix.json — módulos HELPDESK, INCIDENT, PROBLEM, PROJECTS, GMUD, FINANCE, CONTRACTS, SUPPLIERS, ASSETS, TASKS, KB, RISKS, APPROVALS, CONFIG, ACTIVITY_LOG, NOTIFICATIONS, UPLOAD, SUPER_ADMIN, etc.
  • authorize(module, action): carrega utilizador com permissões e verifica par module+action.
  • Escopo (getUserAccessScope / access-scope.js): administradores globais; gestores por departamentos/centros geridos; restantes escopo próprio — aplicado em listagens (ex.: notificações com links, audit log para não-super-admin).

  • In-app, e-mail em muitos fluxos; SSE com token dedicado; preferências PATCH /auth/me/notification-preferences; polling de fallback no layout.

10. Integrações técnicas

  • LDAP: Integration tipo LDAP + LdapService; login /auth/ldap.
  • Azure AD: integração em BD + MSAL; login /auth/azure; GET /auth/azure-config/:tenantSlug.
  • E-mail: MailService / templates; notificações podem disparar envio.
  • IMAP: InboundEmailService + cron para respostas ligadas a tickets (quando configurado).
  • /integrations: atualização e teste de integrações por tipo (CONFIG).

11. Redirecionamentos e legado

  • GET /tasks/:id no backend redireciona 301 para FRONTEND_URL/tasks/:id.

12. Testes e qualidade

  • Backend: Jest, Supertest; config separada para integração (jest-integration.config.js); carga opcional (Artillery).
  • Frontend: Vitest + Testing Library; Playwright para E2E.

13. Implantação e base de dados

Comandos principais (ver também scripts npm em BACKEND/package.json):

  • npm run prisma:generate
  • npx prisma migrate deploy (por schema se necessário)
  • npm run deploy:tenants — migrations em todos os tenants + seed idempotente do catálogo ITIL
  • npm run seed, seed:catalog, seed:catalog:all conforme necessidade
  • Provisionamento de tenant: node src/scripts/provision-tenant.js (quando aplicável)

Scripts operacionais: o directório BACKEND/src/scripts/ contém utilitários de manutenção (migração multi-tenant, seeds parciais, diagnóstico RBAC, integrações, resets controlados, certificados, etc.). Reveja o código e o comentário de cada script antes de executar — não fazem parte do fluxo normal da API em runtime.
